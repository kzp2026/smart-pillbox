# 产品评论知识库智能体 V2 设计规格

状态：待用户审阅
日期：2026-07-16
目标入口：`v2/app.py`
目标应用名：产品评论知识库智能体 V2
目标网址标识：`review-design-agent-v2-kzp2026`

## 1. 目标与成功标准

在不修改、不替换、不下线原网站的前提下，在同一 Git 仓库创建一个独立 V2 Streamlit 应用。V2 使用独立逻辑数据库和私有文件空间，复制原站现有数据与能够展示、下载的历史结果，提供单用户私有登录，并将界面重建为用户截图所示的深色科技控制台风格。

V2 不是原站功能的精简版。成功标准是：

1. 原入口 `app.py`、旧版入口 `app_legacy_current.py` 和 `pages/` 行为保持不变。
2. 原站当前能执行、展示或下载的功能和结果，在 V2 都有明确入口。
3. 未登录时不能读取产品、评论、方案、提示词、图片或历史记录。
4. 所有新数据、历史记录和生成图片在刷新、重启、重新登录后仍可恢复。
5. 同一批数据重复导入不会重复累计评论和需求证据。
6. 每次生成都有独立运行目录、数据库记录、状态、错误和可清理文件。
7. 桌面端接近参考截图；390px 手机端无横向溢出、无被遮挡主操作。
8. 主要功能通过单元、集成、应用和浏览器验收；迁移前后数量与文件哈希可核对。

## 2. 已确认假设

1. 这是单用户私有应用，不开放注册、邀请或多租户。
2. 登录凭据使用 Streamlit Secrets 中的用户名和密码哈希；页面不保存明文密码。
3. V2 继续使用 Streamlit 1.58 和 Python，不把项目改写为 React/Next.js。
4. 数据隔离采用同一 Supabase 项目的私有 `agent_v2` schema，新表不复用原表；这是独立逻辑库，原 schema 保持只读源。
5. 文件持久化采用独立私有 Supabase Storage bucket；本地开发使用独立临时目录适配器。
6. 原站所有现有功能必须保留，包括主知识库流程和旧版十阶段研究流程。
7. DeepSeek 用于真实文本增强；无密钥或请求失败时使用明确标注的离线模板，不伪装成 AI 生成。
8. 图片生成保留 DashScope 与 OpenAI 两套入口；密钥只从 Secrets 读取，页面只显示配置状态和掩码，不提供原始密钥输入框。
9. 付费生成前展示提供商、模型、图片数量和风险提示，必须二次确认；同一运行禁止重复并发提交。
10. 默认视觉验收浏览器为 Chromium，桌面视口 1440×1000、移动视口 390×844。

## 3. 方案选择

### 方案 A：同仓库内模块化 V2（已选）

新增 `v2/` 应用、独立 schema、独立 bucket 和独立 Streamlit Cloud 入口。复用经过测试的分析与生图算法，通过适配器隔离旧代码和新数据模型。

优点：原站零风险、开发速度快、复用率高、可逐项对照迁移。缺点：仓库内会同时维护 V1 与 V2，需要清晰边界。

### 方案 B：直接改造原 `app.py`（不选）

改动少，但会把登录、迁移和新视觉直接叠加在现有 1700 行入口上，容易引入回归，且不能满足“保留原网页”。

### 方案 C：拆成独立仓库或前后端服务（不选）

隔离最彻底，但需要额外部署、接口和运维，交付速度慢，也不符合“同一仓库、新入口”。

## 5. 信息架构与页面

### 5.1 登录页

- 页面只展示产品名、简短说明、用户名、密码、登录按钮和必要错误。
- 登录成功前不初始化知识库、不列出 Storage 文件、不渲染业务页面。
- 连续失败触发冷却；不显示用户名是否存在；不记录明文密码或原始 IP。
- 登录 8 小时无操作自动退出；提供显式退出。

### 5.2 主控制台

桌面端采用参考图的三层结构：

1. 左侧固定配置栏：数据库、文本模型、图片模型、当前产品、运行参数和服务状态。
2. 顶部状态区：品牌、数据库/模型/服务状态、当前时间、用户与退出。
3. 主区域：七阶段流程条、知识库概览、下一步动作、产品资产列表、结果预览。

七个真实阶段：

1. 导入评论资产
2. 需求生成
3. 知识库概览
4. 需求—功能—结构图谱
5. 设计方案与评价
6. 工业设计 Prompt
7. AI 图片生成

### 5.3 结果中心

采用惰性切换，未选中的结果不读取大表或图片：

- 设计方案
- 工业设计 Prompt
- AI 生成效果图
- 需求—功能—结构图谱
- 引用证据与质量报告
- 历史记录
- 下载中心

### 5.4 设置

- 服务配置状态，只显示“已配置/未配置”和密钥末四位掩码。
- 数据迁移：预检、dry-run、正式迁移、数量/哈希核验、报告下载。
- 归档恢复：安全检查、冲突预览、导入确认。
- 数据清理：按运行删除；按产品删除前显示关联数量并二次确认。

## 6. 视觉规格

视觉以用户截图为唯一方向，不沿用当前浅色主题：

- 背景：深海军蓝/近黑，带克制的蓝色空间纹理。
- 强调色：电光蓝、青色、紫色、绿色；橙色只用于警告或“待配置”。
- 面板：低透明深蓝表面、细边框、轻微外发光；不牺牲正文对比度。
- 圆角：主面板 14–18px，按钮 8–10px，状态胶囊 999px。
- 字体：中文优先系统无衬线；标题 600–700，正文 400–500。
- 图标：使用统一的开源 Material Symbols 图标库，不使用 emoji 或自制 SVG。
- 图片：历史/生成产品图片直接作为图库内容；缺图时使用正式空状态，不用假缩略图。
- 视觉资产：独立生成深色背景纹理和小型助手装饰 PNG；不把 CSS 图形冒充图片资产。

响应式要求：

- 1440px：左栏固定，概览与动作两列，图库最多四列。
- 920px：左栏可折叠，主卡片单列或两列。
- 390px：顶部状态压缩，流程为可读的纵向/横向单段导航，表格可切换为卡片，图库单列，所有主要触控目标至少 44px。
- 不允许固定底部元素遮挡内容，不允许主按钮位于不可发现的超长空白之后。

## 7. 技术架构

```text
v2/app.py
  -> auth              登录门禁、超时、失败限流
  -> ui                主题、布局、阶段导航、结果组件
  -> application       导入、分析、生成、历史、删除用例
  -> domain            产品、运行、阶段、资产、迁移模型
  -> adapters
       -> postgres     agent_v2 schema
       -> storage      agent-v2-private bucket / 本地测试目录
       -> legacy       只读读取原 schema、旧运行目录和归档
       -> providers    DeepSeek、DashScope、OpenAI
```

UI 不直接执行 SQL、文件删除或外部 API。应用层负责事务、权限、幂等和状态机；适配器负责具体存储与提供商差异。

## 9. V2 数据模型

所有表位于 `agent_v2` 私有 schema。该 schema 不加入 Supabase Data API 的 Exposed schemas；应用通过服务器端 PostgreSQL 连接访问。表仍启用 RLS 作为纵深防御，并撤销 `anon`、`authenticated` 的默认访问。

核心表：

- `products`：产品资产。
- `comment_batches`：上传批次、来源文件、行数、哈希和状态。
- `comments`：原评论、清洗结果、指纹；`product_id + fingerprint` 唯一。
- `requirements`：需求证据；增加稳定指纹，避免重复导入后重复生成。
- `generation_runs`：用户需求、检索上下文、方案包、质量状态和提供商。
- `pipeline_runs`：完整流程运行、当前阶段、总状态、错误摘要、幂等键。
- `stage_runs`：十个研究阶段的开始/结束、输入哈希、输出摘要和错误。
- `artifacts`：文件类型、Storage 路径、MIME、大小、SHA-256、所属运行和兼容文件名。
- `migration_ledger`：原表/原文件来源标识、目标标识、哈希、迁移时间；保证重复执行不重复写入。
- `login_audit`：仅保存不可逆会话指纹、成功/失败和时间，用于限流；定期清理。

关键约束：

- 业务所有权固定为服务器配置的单用户 ID，页面不能修改。
- 外键完整，产品删除由应用层先列出并清理运行和资产，数据库级联兜底。
- 所有常用筛选、外键和指纹字段建立索引。
- 每个生成运行都有 UUID；Storage 路径为 `runs/{run_uuid}/...`，不再使用共享固定文件名。

## 10. 数据迁移

迁移是显式、幂等、可核验的管理员操作，不在应用启动时自动写入。

### 10.1 迁移来源

1. 原 schema 的 `products`、`comment_batches`、`comments`、`requirements`、`generation_runs`。
2. 原 `generation_runs.result_json/context_json` 中可恢复的方案、Prompt 和状态。
3. 原站可展示或下载的旧版结果：XLSX、CSV、Cypher、JSON、TXT、DOCX、PNG 和 ZIP 中的同类文件。
4. 原 `output/runs` 中可识别的运行目录，以及仓库自带的可展示示例结果。

### 10.2 迁移规则

- 先 dry-run：统计来源数量、冲突、孤立记录、缺失文件和预计容量，不写入。
- 正式迁移时保存原 ID/路径、SHA-256 和来源类型。
- 表记录按来源主键 + 内容指纹幂等；文件按 SHA-256 去重但保留多个兼容引用。
- 无法关联具体生成记录的历史可见文件归入“原站历史快照”，仍可在 V2 查看和下载。
- 迁移不删除、不修改原表和原文件。
- 完成后比较表数量、可见文件数量、总字节数和随机样本哈希，生成迁移报告。

## 11. 关键流程

### 11.1 导入流程

1. 校验扩展名、MIME、文件大小和行列上限。
2. 解析并让用户确认评论列。
3. 清洗、计算文件哈希和评论指纹。
4. 事务写入产品、批次和新增评论。
5. 只为本次真正新增评论生成需求证据，并按需求指纹去重。
6. 保存导入报告和可下载结果。

### 11.2 一键生成流程

1. 创建 `pipeline_run` 和幂等键，锁定重复点击。
2. 按依赖执行旧十阶段；输入哈希未变化且产物有效时可跳过。
3. 每阶段保存状态、日志摘要、产物清单和哈希。
4. 文本模型失败时记录原因并切换到“离线模板”状态。
5. 图片阶段必须先显示计费确认；逐张保存，部分失败不丢失成功图片。
6. 完成后在结果中心打开该运行，并可重新进入。

### 11.3 删除流程

1. 展示将删除的产品、评论、需求、运行和文件数量。
2. 用户输入确认文本。
3. 先标记删除中，删除私有 Storage 对象，再事务删除数据库记录。
4. 文件删除失败时保留可重试状态，不伪装成功。

## 12. 安全与稳定性

- 密码使用带盐的 `scrypt` 哈希，常量时间比较；提供本地生成哈希命令，不要求用户把密码发给 Codex。
- 未登录分支在任何业务查询前 `st.stop()`。
- 登录失败消息统一，限制尝试次数，8 小时无操作退出。
- 数据库连接、DeepSeek、DashScope、OpenAI 和 Storage 密钥只在 Secrets；日志和异常必须脱敏。
- 上传上限由应用配置收紧；ZIP 限制文件数、展开总大小、单文件大小和压缩比，并拒绝路径穿越、绝对路径和符号链接。
- 外部 API 使用超时、有限重试、指数退避和错误分类；不无限重试付费请求。
- 付费生成使用运行级锁和请求幂等检查，按钮提交后立即禁用。
- 历史结果从数据库/Storage 恢复，不依赖 `session_state` 作为唯一来源。
- 不公开 Storage bucket，不在页面输出永久公开 URL。

## 13. 性能

- 登录后首页只加载指标、最近产品和最近运行，不读取所有大表和图片原图。
- 阶段与结果使用条件渲染，避免 `st.tabs` 同时计算隐藏内容。
- 图片先显示缩略图，需要查看或下载时再取原图。
- 列表分页；表格默认最多加载 100 行并提供下载完整文件。
- 常用统计由聚合查询完成；避免逐产品 N+1 查询。
- 热服务目标：首页可交互时间不超过 3 秒；不含用户主动打开的大文件和第三方生成耗时。

## 14. 错误处理与可观察性

- 所有用户错误给出可执行修复建议，不直接显示连接串、密钥或完整堆栈。
- 每次运行有短 ID、状态、阶段耗时和最后错误。
- 数据库/Storage 不可用时禁止悄悄回退到共享本地库；进入只读错误页。
- 部分生成成功时明确显示“部分完成”和失败资产，可只重试失败项。
- 迁移、删除和付费生成均记录审计事件。

## 15. 测试策略

严格采用测试驱动：先新增失败测试，再写最小实现，再重构。

### 单元测试

- 登录哈希、会话超时、失败冷却和脱敏。
- 文件/ZIP 安全规则。
- 评论与需求双层去重。
- 运行状态机、幂等键和删除计划。

## 17. 代码风格

- Python 采用类型标注、短函数、显式依赖注入；UI、用例、数据库和提供商分层。

- 领域状态使用枚举或明确常量，不使用散落魔法字符串。

## 18. 边界

始终执行：

- 保留原站，先测试再实现，所有删除和付费操作二次确认。
- 所有数据和文件按运行隔离并可追溯。
- 迁移先 dry-run，再由登录用户显式执行。

实施中需要额外确认：

- 改动原有数据库表、删除原文件、提高付费生成数量上限。
- 增加新的收费服务或切换到独立 Supabase 项目。

绝不执行：

- 把密钥、密码或连接串写入代码、测试夹具、日志或 Git。
- 未经确认删除原站、原表、原文件或历史结果。
- 在未登录页面暴露数据、图片 URL 或可猜测的 owner ID。
- 用固定共享文件名覆盖其他运行的效果图。
