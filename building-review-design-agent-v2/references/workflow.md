# V2 网页创建与持续扩展流程

## 1. 业务规格固定

先把用户语言转成可验收结果：保留原网页；同仓库新增 V2；使用独立入口和网址；独立私有 schema 与资产空间；复制而非搬迁现有数据；单用户私有登录；完整保留结果、效果图、下载和旧十阶段能力；视觉对齐用户截图。

关键方向存在歧义时一次性汇总提问。用户选择“全部按推荐”后，将结论写入设计规格，不反复询问已确认事项。

## 2. 现状与视觉审计

只读检查线上网页、仓库、数据/资产入口、未提交改动和测试。记录桌面、移动、登录、空数据、已有数据、错误态与主要用户旅程。把用户截图视为视觉方向，但把仓库和真实数据视为功能真值。

先列问题再实施：功能断链、数据丢失风险、权限泄露、重复生成、错误反馈、响应式、可访问性、性能和视觉差异都要覆盖。公开错误必须经过统一脱敏层，禁止直接把数据库或第三方异常原文插入页面。

## 3. 隔离原站

建立 `tests/v2/test_original_freeze.py`，冻结原入口与原页面。所有新代码进入 `v2/`，新测试进入 `tests/v2/`，新部署以 `v2/app.py` 为 Main file path。除非用户明确授权，不把登录、迁移或新视觉叠到 `app.py`。

## 4. 架构与数据空间

按 UI、application、domain、adapter/provider 分层。UI 不直接执行 SQL、删除或外部 API。使用 `agent_v2` 私有 schema、固定 `V2_OWNER_ID` 和独立私有资产存储。先定义表、幂等键、外键、索引、RLS/revoke 和删除策略，再实现适配器。

数据库演进使用新增的编号 SQL migration；不要改写已在生产执行过的 migration。部署顺序保持向后兼容：先扩展 schema，再部署兼容代码，再迁移数据，最后清理废弃路径。

## 5. 单用户私有登录

登录用户名和 scrypt 密码哈希只来自 Secrets。登录成功前停止业务初始化；失败消息不泄露用户名是否存在；5 次失败冷却 15 分钟；8 小时无操作退出；提供显式退出。不要为了“保存筛选”或类似功能引入注册、Supabase Auth 或多租户 user_id。

## 6. 功能完整迁移

逐项使用设计规格中的功能矩阵，把主知识库流程、旧十阶段、历史结果、效果图、归档、下载、产品管理和提供商入口映射到 V2。适配旧文件名和历史快照，不伪造缺失数据。真实数据未迁移时显示正式空状态。

历史复制流程必须是 dry-run → 登录用户确认 → apply → 再次 apply 验证幂等 → 数量、总字节和 SHA-256 核验。来源数据库和来源文件始终只读。

## 7. 测试驱动实现

每项功能先添加会因缺失行为而失败的测试：

- 纯规则和安全逻辑：单元测试。
- 仓库、迁移、Storage、幂等和恢复：集成测试。
- 登录、导航、表单、下载和错误态：Streamlit AppTest。
- 真实交互、响应式和视觉：Chromium 浏览器验收。

看到测试按预期失败后写最小实现；通过后再重构。错误修复先写复现测试。外部收费 API 使用假实现验证参数与确认流程，只有最终受控验收才调用真实服务。

## 8. 切页性能与百炼 Key 入口

Streamlit 任何控件交互都会重跑脚本，不能在全局页头逐表查询。把工作台计数合并为一条 `workspace_snapshot()` 聚合查询，产品、运行与同一运行详情经过 30 秒只读缓存复用；所有写入成功或失败后，只要持久状态可能变化就立即按仓库作用域失效，退出时清空。运行时背景、Logo 和助手使用优化 WebP，原 PNG 仅保留为视觉源；三项 data URI 合计小于 900 KB。

入口脚本的普通全局变量不会跨 Streamlit 重跑保留。仓库、Storage、登录守卫和视图缓存统一放在 `v2/application/runtime_state.py`；主题模块用进程缓存只编码一次 WebP。AI 效果图页先用 `include_data=False` 打开元数据并完成页面切换，预览按钮触发后仅批量读取 `image/` 资产；PostgreSQL blob 读取必须使用一个连接和一条 `IN (...)` 查询。桌面与移动导航的验收预算均为 5 秒以内，大图预览加载时间单独计算并明确显示。

会话使用 `v2_active_product` 隔离产品上下文。导入或需求页填写新产品时清除上一运行选择；结果页通过 `target_product` 过滤运行，历史页先选择产品再列出该产品记录。不得删除或覆盖其他产品历史，只做默认隐藏。

阿里云百炼效果图 Key 的可见入口固定在侧栏，点击后进入“设置与迁移”的“百炼效果图 Key 配置”。页面只展示 `V2_IMAGE_PROVIDER`、`V2_IMAGE_MODEL`、`V2_IMAGE_API_KEY` 占位模板和 Streamlit 应用管理链接；Key 仍只写入新应用 Secrets，不进入页面表单、session、数据库、日志或 Git。DeepSeek 已配置时不重复要求填写。

## 9. 深色控制台视觉实现

从视觉 token、布局骨架、可复用组件到页面逐层实现。正式背景、品牌标记和助手使用真实 raster 资产。保留 Streamlit 的可操作语义、可见标签、键盘焦点和移动导航。

每次视觉改动同时检查 1440×1000 与 390×844；比较完整页面而不是孤立组件。检查文字层级、间距、颜色、边框、图片质量、溢出、触控目标、空/有数据状态和浏览器控制台，并更新 `design-qa.md` 及 QA 截图。原生上传、下载、链接、密码切换、代码复制、主次按钮及其 hover、disabled、focus 状态不得回落为白底；固定助手在手机宽度隐藏，不能遮挡操作。

## 10. 验证与发布

先运行受影响测试，再运行 Skill 契约校验、全部 V2 测试和仓库全量测试。启动 V2 与原站冒烟时必须带 `--global.developmentMode=false`。发布新 Streamlit 应用时只配置 V2 Secrets，不修改原应用入口、Secrets 或网址。

Streamlit Cloud 可能只重新执行入口脚本而继续复用进程中上一版本的已导入模块。新增只读领域类型时，`v2/app.py` 不得在顶层直接导入该新类型；入口按属性协议消费仓库返回值，并提供本地零值回退。发布前用回归测试同时覆盖“入口无新增类型导入”和“旧仓库对象没有新方法”两种状态，避免热更新期间出现整站 `ImportError`。

同一 Supabase 项目部署第二个 Streamlit 应用时，优先把原应用中已验证可用的 `PRODUCT_KB_DATABASE_URL` / `DATABASE_URL` 值复制为 `V2_DATABASE_URL`，并依靠 `V2_SCHEMA = "agent_v2"` 隔离数据。若改用 Shared Transaction Pooler，连接串必须从 Supabase **Connect** 面板完整复制，用户名形如 `postgres.<project-ref>`、端口为 `6543`，使用数据库密码而不是 V2 登录密码。不得为修复单个新应用而直接重置数据库密码；确需重置时必须同步更新所有关联应用。

生产迁移前先确认登录、数据库、Storage、导航和零付费生成路径。迁移后核对计数与哈希，再验证历史打开、下载、安全恢复和受控付费确认。最后重新访问原网址。

## 11. 后续新增功能时同步 Skill

每个新增页面、字段、表、provider、环境变量、导航、产物、部署步骤或验收门都属于 Skill 契约变化。实现它的同一任务中：

1. 更新 `references/project-contract.md` 的当前真值。
2. 更新本流程中受影响的顺序或边界。
3. 更新 `references/verification.md` 的测试和证据。
4. 扩展 `scripts/verify_contract.py`，让遗漏能自动失败。
5. 运行校验并在交付说明中列出 Skill 同步内容。

如果只是内部重构且外部行为、路径、约束和验证命令均未变化，仍运行契约校验，并明确记录“无需改变 Skill 契约”的依据。
